# SPEC.md — 三藏原文解析（AI 輔助研經工具）

> 工作代號：**TIPITAKA-LENS**（暫定）
> 對標專案：bible-recovery-analyzer
> 核心差異：以 LLM（Claude Sonnet）補上「漢譯逐字解析」這個本來不存在的標註層，並嚴守「權威資料 / 生成內容」分離。

---

## 0. 一句話定位

把**巴利語三藏**的逐字語法解析（吃 DPD 現成標註）當分析骨幹，把**漢譯阿含**並排對照當漢文入口，再用 **Sonnet** 在「有真實資料可依據」的前提下，生成漢譯逐字解釋、章節概要、概念說明與語意搜尋。

對標專案的價值來自「消費別人標好的 SBLGNT/OSHB」。我們的價值來自：**巴利語照樣消費現成標註（DPD），漢譯這條死路則用 LLM grounding 打通。**

---

## 1. 核心設計原則：三層資料分離

整個系統的成敗都壓在這條界線上。違反它，產品就會在佛教社群裡失去信任。

### L1 — 權威層（Deterministic，永不由 LLM 生成）
- 經文原文（巴利、漢譯）
- 逐字詞形 / 字根 / 變化形分析（DPD）
- 平行經文對齊關係（SuttaCentral parallels）
- 字典詞條本文（DPD / PTS / DPPN）

**規則：L1 一律來自外部權威資料源，標明出處，可追溯，不可被模型改寫。** UI 上 L1 內容不得與 L2 混色、混排到無法分辨。

### L2 — 生成層（LLM，必須 grounded on L1）
- 漢譯逐字白話對照（**以對齊的巴利 DPD 真實語法資料為依據**生成）
- 章節概要 / 研經卡
- 概念解釋、譯本用字差異說明
- 語意搜尋的查詢理解與結果重排

**規則：L2 的每一次生成都必須把對應的 L1 資料塞進 prompt 當依據（RAG）。模型不得在沒有 L1 依據時自由發揮。所有 L2 內容在 UI 上標記為「AI 輔助生成」並附上它依據的 L1 來源。**

### L3 — 互動層（LLM，對話式）
- 問經文、追問語法、要求比對譯本
- 同樣 grounded，同樣標示，同樣附來源

> 一句判準：**凡是「事實」走 L1，凡是「解釋」走 L2/L3，且解釋永遠掛在事實上。** 模型可以解釋字根為什麼這樣變化，但「這個字的字根是什麼」必須來自 DPD，不准模型自己講。

## 2. 資料源（L1 權威層）

| 資料 | 來源 | 形式 | 授權 |
|---|---|---|---|
| 巴利經文（分段對齊） | SuttaCentral `bilara-data` | JSON / API | 公共領域底本 |

## 3. LLM 能力地圖（Sonnet 在哪裡、做什麼、靠什麼 grounding）

| 功能 | 模型角色 | Grounding 依據（必填） | 若無依據 |
|---|---|---|---|
| 漢譯逐字解析 | 生成白話對照 + 用字說明 | 對齊的巴利經 + 該段 DPD 詞形/字根 | 退回「僅顯示漢譯原文，無逐字解析」 |
| 章節概要 | 摘要 | 該章 L1 全文 | 不生成 |
| 研經卡 | 結構化重點卡 | 該段 L1 + DPD 詞條 | 不生成 |
| 概念解釋（如「無明」「緣起」） | 解說 + 出處索引 | 檢索到的相關經文段落 | 標示「查無經證」 |
| 語意搜尋 | 理解查詢 → 重排結果 | 全文索引候選結果 | 退回關鍵字搜尋 |
| 對話研經 | 問答 | 當前頁 L1 + 檢索段落 | 拒答並說明 |

**為什麼漢譯逐字解析不是幻覺：** 漢譯阿含本身沒有逐字標註，但 SuttaCentral 已把它對齊到對應的巴利經，而巴利經有 DPD 的真實字根/詞形資料。模型是「拿巴利的真實語法事實，去解釋漢譯譯者選了哪個字」，不是憑空編造漢字的字源。這是有依據的對照，不是發明。

## 4. 頁面與功能規格

### 4.1 研經頁 `/study/{collection}/{sutta}`（核心頁，對標 study 頁）
三欄式（桌機）/ 可切換頁籤（手機）：
- **欄 A：巴利原文**，逐字可點，點字浮出 DPD 詞條（字根、詞形碼、變化形、複合詞拆解）— 純 L1
- **欄 B：漢譯阿含對照**，逐字白話由 L2 生成，每個解釋標「AI」徽章並可展開看它依據的巴利字
- **欄 C：研經卡 + 章節概要**（L2，附 L1 來源清單）

### 4.2 搜尋 `/search`
- 經文 / 字詞 / lemma / 全文（=對標專案四種）→ 全部 L1，走本地索引
- **新增「語意搜尋」**（L2）：自然語問題 → 模型理解 → 在 L1 索引候選上重排 → 標示為 AI 輔助
- 兩種搜尋結果分區顯示，不混淆

### 4.3 字典 `/lexicon`
- DPD 詞條（L1）+ 該字在全藏的出現位置（L1 索引）
- 「這個字在不同語境的用法差異」摘要（L2，grounded on 實際出現的句子）

### 4.4 對話研經（新功能，L3）
- 頁內側邊欄，預設帶入當前經文 context
- 系統 prompt 強制：只依據傳入的 L1 段落作答、附引用、不確定就說不知道、不裁決宗派爭議

## 5. 系統架構

延續對標專案的**靜態站**路線（GitHub Pages 可托管），LLM 走前端直連 Anthropic API（在 Artifact / 前端環境由平台代管金鑰，不在程式碼放金鑰）。

```
靜態前端 (GitHub Pages)
  ├─ 預建資料 (build-time)
  │    ├─ 巴利經文 + DPD 標註 → 預先 join 成每章 JSON
  │    ├─ 平行對照表 (巴利↔阿含)
  │    └─ 全文 / lemma 索引 (本地搜尋用)
  └─ 執行期 (runtime)
       ├─ L1 讀預建 JSON（零 LLM、可離線）
       └─ L2/L3 呼叫 Sonnet，prompt 內嵌對應 L1 段落 (RAG)
```

**設計重點：L1 全部 build-time 預建成靜態 JSON。** 使用者翻經文、查字、看標註完全不需要 LLM，速度快、可離線、零成本。只有「要解釋」時才呼叫模型。這同時也是成本控制策略——絕大多數頁面瀏覽不觸發 API。

### 5.1 生成內容快取
L2 內容（章節概要、漢譯逐字解析）對同一段經文是穩定的 → **build-time 預生成 + 人工校稿後固化成靜態檔**，而非每次即時呼叫。即時 LLM 只留給 L3 對話與語意搜尋。這樣可信度更高（可校稿）、更省、更快。

## 6. 資料模型（JSON Schema 草案）

### 6.1 經文段落
```json
{
  "segment_id": "mn1:1.1",
  "pali": { "text": "...", "tokens": [
    { "surface": "evaṃ", "lemma": "evaṃ", "dpd_id": 12345,
      "root": "...", "morph": "ind", "compound": null }
  ]},
  "chinese_parallels": [
    { "source": "CBETA", "ref": "T01n0026_p0421a", "text": "..." }
  ],
  "sources": ["suttacentral:mn1", "dpd-db:vX"]
}
```

### 6.2 L2 生成內容（永遠帶 grounding 與標記）
```json
{
  "segment_id": "mn1:1.1",
  "type": "chinese_gloss",
  "generated_by": "claude-sonnet-4-6",
  "grounded_on": ["mn1:1.1.pali.tokens", "cbeta:T01n0026..."],
  "content": "...",
  "reviewed": false,
  "review_status": "draft|approved"
}
```
`reviewed:false` 的內容在 UI 標「未校稿」。固化進靜態檔的應為 `approved`。

## 7. LLM 防護欄（這節決定產品死活）

1. **不准無依據生成**：L2/L3 prompt 一律附 L1；缺依據就退化降級，不硬生。
2. **教義中立**：上座部 / 大乘 / 藏傳對同一概念詮釋常不同。模型**呈現並標明各傳統說法，不裁決對錯、不以單一宗派詮釋當定論**。
3. **字根/詞形不准模型講**：一律來自 DPD。模型只負責「解釋」DPD 給的資料，不負責「提供」它。
4. **強制引用**：每段 L2/L3 輸出附其依據的 L1 segment_id / 經號。
5. **可校稿**：固化前的生成內容走人工審核流程（`draft → approved`）。
6. **明確標示**：所有生成內容帶「AI 輔助」徽章，與 L1 視覺可區分。
7. **誠實降級**：查無經證 / 不確定 → 明說，不補洞。

## 8. V1 範圍 / 非範圍

**V1 做：**
- 巴利四部尼柯耶中**先挑一部**（建議中部 Majjhima Nikāya，篇幅適中、教義代表性高）
- 該部全經：巴利逐字（DPD）+ 漢譯中阿含對照 + 預生成概要/研經卡（已校稿）
- 四種 L1 搜尋 + 本地索引
- DPD 字典頁
- L3 對話研經（限當前經 context）

**V1 不做：**
- 全藏（先驗證一部的資料管線與校稿流程）
- 即時 L2 大量生成（先預生成 + 校稿）
- UGC / 使用者註解
- 藏文、梵文層
- 商業化 / 付費

## 10. 風險清單

| 風險 | 嚴重度 | 緩解 |
|---|---|---|
| LLM 幻覺佛法/捏造字根 | 高 | 三層分離 + 強制 grounding + 字根只走 DPD + 人工校稿 |
| 宗派爭議引發信任崩塌 | 高 | 教義中立、呈現多說、不裁決 |
| 漢譯逐字解析品質參差 | 中 | 以巴利對照為錨、標 AI、可校稿、缺依據就降級 |
| 即時 API 成本失控 | 中 | L1 全靜態、L2 預生成固化、即時只留 L3 |
| 授權違規（誤商業化） | 中 | 合規清單 + 全站非商業 |
| 對齊資料不完整（某些漢譯無巴利平行） | 中 | 該段退化為「僅原文 + 概要」，不硬生逐字 |

## 11. 里程碑

1. **資料管線 PoC**：跑通「一部經」的 巴利+DPD join、阿含對齊、靜態 JSON 預建
2. **研經頁原型**：三欄顯示 + 點字查 DPD（純 L1，先不接 LLM）
3. **L2 預生成 + 校稿流程**：概要/漢譯逐字，落地 `draft→approved`
4. **L3 對話 + 語意搜尋**：接 Sonnet，掛防護欄
5. **一部完整上線**：中部全經，驗證可信度與成本
6. **擴充決策點**：再決定是否推全藏 / 開放更多功能
